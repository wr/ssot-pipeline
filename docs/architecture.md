# ssot-pipeline architecture

## The three pillars (v1)

### 1. Config-as-code

All "magic strings" — plan marker, Linear MCP URL/transport, workflow state names, approval emoji, approved user IDs, project→repo mapping — live in **`config/pipeline.json`** at the repo root.

- The Worker imports it at build time (`import config from "../../config/pipeline.json"`) and exposes it at `GET /config`.
- Workflows fetch the config at job start: `curl -sS https://<worker>/config | jq -r '.plan_marker'`.
- Change a value: edit the JSON, commit, redeploy the Worker. All consumers see it on the next event.

**Why Worker-served and not raw GitHub:** the repo is private; raw.githubusercontent.com requires auth tokens that don't cross repos. The Worker is already the runtime spine and the config is public (no secrets in it).

### 2. Trace IDs

Every Linear webhook event gets an 8-character trace ID (`crypto.randomUUID().slice(0, 8)`) generated by the Worker. It rides through:

| Layer | Where the trace appears |
|---|---|
| Worker logs | `trace=XXXXXXXX received Issue.update for W-NN` |
| GitHub repository_dispatch | `client_payload.trace_id` |
| Workflow first step | `echo "🔗 trace=XXXXXXXX issue=W-NN"` |
| Linear comments by the bot | trailing `_(trace: XXXXXXXX)_` |

Debugging shrinks to: copy a trace ID from any one place, grep across all three systems, see the entire causal chain.

### 3. Mandatory verification + Stuck state

Every workflow ends with an `if: always()` verification step that asserts the expected post-conditions via Linear and GitHub APIs:

| Workflow | Asserts |
|---|---|
| `linear-pickup` | issue state == `Plan Review` AND ≥1 comment starting with the plan marker |
| `linear-implement` | issue state == `In Review` AND ≥1 GitHub PR attachment on the issue |
| `pr-review` | ≥1 review submitted by `claude[bot]` on the PR |

If the assertion fails: the step posts a diagnostic comment with the trace ID, flips the issue to a `Stuck` workflow state, and exits 1 (red X). No silent successes.

The `Stuck` state must exist in the Linear team workflow (type: Started). The verification step warns but doesn't crash if it's missing.

## Component map

```
Linear webhook ──HMAC verify──▶ Cloudflare Worker
                                  │
                          generates trace_id
                                  │
                          looks up project_to_repo
                                  │
                                  ▼
                      GitHub repository_dispatch
                                  │
                                  ▼
                       Target repo's ssot.yml stub
                                  │
                  routes by event_name / event.action
                                  │
                                  ▼
            ssot-pipeline reusable workflow (pickup / implement / pr-review)
                                  │
                                  ▼
                claude-code-action runs claude -p with MCP
                                  │
                                  ▼
                       Verification step asserts outcome
                                  │
              ┌───────────────────┴──────────────────┐
              ▼                                      ▼
       ✓ outcome matches                  ✗ outcome missing
       (green)                            → diagnostic comment
                                          → flip to Stuck
                                          → exit 1 (red)
```

## Auth + identity model

| Identity | Used for | How |
|---|---|---|
| `@claude` (Linear) | All comments, reactions, state changes from the loop | Linear OAuth app token with `actor=app` (`LINEAR_APP_TOKEN`) |
| `claude[bot]` (GitHub) | Reviews + (intermittently) commits | Anthropic-published GitHub App installed via `/install-github-app` |
| `github-actions[bot]` (GitHub) | Default git push from `linear-implement` | `GITHUB_TOKEN` (the default workflow token) |
| Worker → GitHub dispatch | `repository_dispatch` API calls | Fine-grained PAT (`GITHUB_DISPATCH_TOKEN`) with Contents+Actions write |

Open identity gap: `GITHUB_TOKEN` can't push workflow file changes — it's forbidden regardless of `permissions:`. Tracked in W-78.

## Where state lives (and where it doesn't)

| State | Lives in | Lifetime |
|---|---|---|
| Issue status, comments, reactions, attachments | Linear (canonical) | Forever |
| Pipeline config | `config/pipeline.json` (canonical) → Worker (cached at build time) | Until next Worker deploy |
| Secrets (HMAC keys, OAuth tokens, PATs) | Wrangler secrets + GitHub repo secrets | Until rotated manually |
| PR contents, branches, reviews | GitHub | Forever (in repo history) |
| Workflow run logs | GitHub Actions | 90 days (default) |
| Trace IDs | Per-event, ephemeral (just a correlation handle) | One run |

The Worker keeps **no persistent state** of its own. KV is not used. Everything is request-scoped or sourced from the imported config.
