The gap

Autonomous agents went fire-and-forget. The bill and the blast radius showed up later.

The frontier is swinging back to “a senior engineer stays in the loop.” We started there — and kept the infrastructure down to a single free function.

How it works

Delegate an issue. Get a reviewed PR. Merge when you’re ready.

Delegate a Linear issue to @claude

A webhook hits your Worker, which mints a trace ID and dispatches a GitHub Action. The issue flips to Planning within seconds.

Claude posts a plan

It reads the issue and your codebase, then posts an actionable plan as an in-session elicitation. The issue moves to Plan Review.

You approve — the first gate

Reply approve and it builds. Reply with changes and it re-plans. No code is written until you say go.

Claude implements & opens a PR

It branches, writes the code, and opens a pull request — Closes W-XX so Linear back-links automatically.

Claude reviews its own PR

A separate reviewer identity approves or requests changes. Requested changes auto-fix and re-review, capped, until clean.

You merge — the gate that matters

The sign-off is your merge under branch protection. Linear closes the issue on merge. Nothing ships without this step.

Quickstart

Wire a repo up with one command.

init-target-repo
# point the loop at a repo + its Linear project
$ ./bin/init-target-repo.sh ./my-app linear.app/you/project/my-app
✓ installed ssot.yml workflow
✓ set repo secrets
✓ mapped project → repo, redeploying Worker…
→ delegate an issue to @claude to test
# trace=a1b2c3d4 flowing through all three systems
  1. Deploy the Worker — Cloudflare’s free tier. One command, no servers.
  2. Install the GitHub Apps — one to act, one to review.
  3. Point your Linear webhook at the Worker URL.
  4. Run init per repo — the script does secrets, workflow, and routing end-to-end, idempotently.

Then: delegate a Linear issue → watch the trace flow → merge when you’re ready.
